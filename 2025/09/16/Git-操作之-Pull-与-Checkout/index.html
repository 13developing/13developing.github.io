Git 操作之 Pull 与 Checkout

从前使用 git 都是个人简单项目，所以 git 操作比较少。本次做项目，博主将每一次课程的内容都创建了一个分支，故每次针对性的学习章节内容就可以从这些分支中获取内容。

创建一个新的项目之后，首先是配置远程仓库。但是配置了项目之后，并不知道项目的分支情况，故需要一次fetch，IDEA 中有按键**Fetch All Remotes** ，其意思是：与已配置的远程仓库（通常是 origin）进行通信，获取它上面所有分支的最新状态和信息，并更新到本地仓库记录中。

在获取了All Remotes 之后，如何将分支传输回本地呢？这里是有两个选择的：Checkout 和 Pull.

首先先说答案，对于第一次拉取该分支，是应该Checkout。

  • 当你执行 Checkout 一个远程分支（比如 origin/chapter-03）时，Git 会做两件事：
    1. 以远程分支为蓝本，创建一份一模一样的本地副本。这个本地副本的名字通常就是 chapter-03（会自动去掉 origin/ 前缀）。
    2. 立即把你桌上的东西换成这本书的副本。你的工作目录里的文件会立刻变成 chapter-03 分支的内容。
  • 关键：Checkout 主要是一个切换和创建本地分支的操作。它的目的是让你开始基于某个分支进行工作

而Pull是在什么情况下使用呢？

  • 它的使用场景是：你已经有本地副本了（比如你之前已经 checkout 了 chapter-03），现在你想更新它。

即在你已经 Checkout 之后，远程该分支更新了一次，你先获取最新的该分支，则可使用 Pull,他会把最新的和你的本地副本进行合并(Merge)

那如果没有本地副本的时候，Checkout和Pull是否是一样的呢？

答案是否。当无本地分支副本的时候，进行Pull时:

  1. Git 的行为（它会尝试这样做）：
    • “你想把远程 origin 的 chapter-03 分支拉取下来。”
    • “但是我发现你本地没有一个叫 chapter-03 的分支让我合并进去啊？”
    • “算了，那我就在你当前所在的分支（比如 main）上，尝试把 chapter-03 的内容合并过来吧。”
  2. 结果： 这会导致远程 chapter-03 分支的所有内容被合并到了你当前的分支（如 main），而不是创建一个新的 chapter-03 分支。这会把你的当前分支搞得一团糟，完全不是你想要的结果。
